# Styll — Local Development Setup Guide

> **Generated by auditing the actual repo files.** Every command and port
> number is derived from `supabase/config.toml`, `package.json`, `.env.example`,
> `apps/web/CLAUDE.md`, and `src/proxy.ts`. Not generic advice.

---

## ⚠️ HEADS UP — Unusual Stack Choices

Read these before starting — they affect tooling:

| Concern | Detail |
|---------|--------|
| **Next.js 16.2.4** | Bleeding-edge release. Uses `src/proxy.ts` + `export proxy()` instead of the standard `src/middleware.ts` + `export middleware()`. Do **not** rename it. |
| **React 19.2.4** | Some older third-party libraries may emit warnings. Do not downgrade — the codebase targets React 19. |
| **Tailwind CSS v4** | Breaking changes from v3. Config is handled via `@tailwindcss/postcss` in PostCSS — there is **no `tailwind.config.js`**. Do not create one. |
| **No `.nvmrc`** | The repo has no Node version pin. Use **Node.js ≥ 20** (LTS). |
| **`type-check` script missing in `apps/web`** | The root `package.json` defines `"type-check": "pnpm --filter web type-check"` but `apps/web/package.json` has no `type-check` script. Running `pnpm type-check` from the root will fail. Use `pnpm --filter web exec tsc --noEmit` directly instead. |
| **Subdomain routing in local dev** | The proxy maps `*.localhost:3000` to tenant routes. You access tenant dashboards via `marco-dashboard.localhost:3000`, not `/dashboard`. See Section 3. |

## Section 1 — Prerequisites

### Node.js ≥ 20 (LTS)

```bash
# via Homebrew + nvm (recommended)
brew install nvm
nvm install 20
nvm use 20
node --version  # should be v20.x.x or higher
```

### pnpm 9.0.0 (exact — declared in `packageManager` field)

```bash
npm install -g pnpm@9.0.0
pnpm --version  # must be 9.0.0
```

> Using a different pnpm version may cause lockfile conflicts.

### Docker Desktop

Required to run Supabase locally. Supabase CLI spins up Postgres, Auth, Studio, Inbucket, and Storage as Docker containers.

```bash
# Download from: https://www.docker.com/products/docker-desktop/
# Or via Homebrew:
brew install --cask docker
```

After installation, **open Docker Desktop and keep it running** before any `supabase` commands.

### Supabase CLI

```bash
brew install supabase/tap/supabase
supabase --version  # any recent version works (≥ 1.200)
```

## Section 2 — First-Time Setup (run once)

### Step 1 — Clone and install dependencies

```bash
git clone https://github.com/tvwebspecialist/Styll.git
cd Styll
pnpm install
```

### Step 2 — Create your `.env.local`

Create `apps/web/.env.local` (this is where Next.js reads it from):

```bash
cp .env.example apps/web/.env.local
```

Then open `apps/web/.env.local` and fill in the values using the template below.

#### `.env.local` template with safe local defaults

```bash
# ── Supabase (local) ──────────────────────────────────────────────────────────
# These values come from `supabase start` output (Step 4).
# Run supabase start first, then paste the keys here.
NEXT_PUBLIC_SUPABASE_URL=http://localhost:54321
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=<anon_key_from_supabase_start>
SUPABASE_SECRET_KEY=<service_role_key_from_supabase_start>

# ── App ───────────────────────────────────────────────────────────────────────
NEXT_PUBLIC_APP_URL=http://localhost:3000
NEXT_PUBLIC_APP_NAME=Styll

# ── Subdomain routing ────────────────────────────────────────────────────────
# Required for tenant URL parsing in proxy.ts.
# In local dev, *.localhost:3000 is parsed automatically.
# Leave this unset locally (proxy.ts defaults to 'styll.it' but local
# subdomain detection uses the .localhost:3000 pattern separately).
# NEXT_PUBLIC_ROOT_DOMAIN=styll.it

# ── Resend (email) ────────────────────────────────────────────────────────────
# Required for team invitations. Without it, inviting team members fails.
# Free tier at https://resend.com — no credit card needed.
# Leave as placeholder to skip email features during local dev.
RESEND_API_KEY=re_placeholder_not_used_locally

# ── Stripe (optional — billing not in MVP) ───────────────────────────────────
# STRIPE_SECRET_KEY=sk_test_...
# NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_test_...
# STRIPE_WEBHOOK_SECRET=whsec_...

# ── Twilio (optional — SMS/WhatsApp) ─────────────────────────────────────────
# TWILIO_ACCOUNT_SID=AC...
# TWILIO_AUTH_TOKEN=...
# TWILIO_PHONE_NUMBER=+39...

# ── Sentry (optional — error monitoring) ─────────────────────────────────────
# SENTRY_DSN=https://...
# NEXT_PUBLIC_SENTRY_DSN=https://...

# ── PostHog (optional — analytics) ───────────────────────────────────────────
# NEXT_PUBLIC_POSTHOG_KEY=phc_...
# NEXT_PUBLIC_POSTHOG_HOST=https://app.posthog.com
```

> **Note:** `NEXT_PUBLIC_ROOT_DOMAIN` is not in `.env.example` but is read by
> `src/proxy.ts`. For local dev you do **not** need to set it — the proxy
> separately handles `*.localhost:3000` via the `getSubdomain()` function.

### Step 3 — Start Docker Desktop

Open Docker Desktop and wait for it to show "Docker Desktop is running" in the menu bar. All subsequent Supabase commands require Docker.

### Step 4 — Start Supabase and get your local keys

```bash
supabase start
```

Expected output (ports from `supabase/config.toml`):

```
Started supabase local development setup.

         API URL: http://localhost:54321
     GraphQL URL: http://localhost:54321/graphql/v1
  S3 Storage URL: http://localhost:54321/storage/v1/s3
          DB URL: postgresql://postgres:postgres@localhost:54322/postgres
      Studio URL: http://localhost:54323
    Inbucket URL: http://localhost:54324
      JWT secret: super-secret-jwt-token-with-at-least-32-characters-long
        anon key: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...  ← paste as NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
service_role key: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...  ← paste as SUPABASE_SECRET_KEY
```

**Now paste those two keys into `apps/web/.env.local`:**

```bash
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=eyJhbGciOiJIUzI1NiIs...   # anon key
SUPABASE_SECRET_KEY=eyJhbGciOiJIUzI1NiIs...                    # service_role key
```

> If you've already run `supabase start` before, run `supabase status` to
> retrieve the keys again without restarting.

### Step 5 — Apply all migrations

```bash
supabase db push
```

This applies all 10 migrations in order:

| # | File | What it creates |
|---|------|----------------|
| 1 | `20260425000001_profiles.sql` | `profiles`, `staff_members`, tenants, subscriptions, appointments, clients, loyalty, payments, etc. |
| 2 | `20260425000003_avatars_storage.sql` | Storage bucket for avatars |
| 3 | `20260425000005_superadmin.sql` | `is_superadmin` flag + admin utilities |
| 4 | `20260426000001_admin_v2.sql` | Admin panel v2 schema |
| 5 | `20260427000001_portfolio.sql` | `portfolio_photos` table |
| 6 | `20260430000001_team_invitations.sql` | `team_invitations` table (required for staff invite feature) |
| 7 | `20260501000001_client_analytics.sql` | Client analytics views |
| 8 | `20260501000002_client_analytics_cron.sql` | Cron jobs for analytics |
| 9 | `20260507105244_backfill_staff_locations.sql` | Backfills `staff_locations` data |
| 10 | `20260601000001_client_import_jobs.sql` | CSV import jobs table |

Verify everything applied:

```bash
supabase migration list
# All 10 should show ✓ applied
```

### Step 6 — Run seed data

The seed creates a demo tenant ("Marco's Barber") with staff, services, clients, appointments, and loyalty data — everything you need to see the dashboard working.

```bash
# The seed.sql is an orchestrator that sources 6 sub-files
psql postgresql://postgres:postgres@localhost:54322/postgres -f supabase/seed.sql
```

> **Note:** The seed uses `\i` directives (relative paths). Run this command from
> the **repo root**, not from inside `supabase/`.

After seeding:
- Tenant slug: `marco`
- Dashboard URL: `http://marco-dashboard.localhost:3000`
- Landing/booking URL: `http://marco.localhost:3000`

### Step 7 — Create a superadmin account

1. Register a new account via `http://localhost:3000/register`
2. Open Supabase Studio at `http://localhost:54323`
3. Navigate to **Table Editor → profiles**
4. Find your user row and set `is_superadmin = true`

Or via SQL:

```sql
-- Run in Studio > SQL Editor, or via psql:
UPDATE public.profiles
SET is_superadmin = true
WHERE email = 'your@email.com';
```

The admin panel is then accessible at `http://localhost:3000/admin`.

```bash
# Terminal 1 — Database (if not already running)
supabase start

# Terminal 2 — Next.js dev server
pnpm dev
# or: pnpm --filter web dev
```

| Service | URL | Notes |
|---------|-----|-------|
| **Next.js app** | `http://localhost:3000` | Login, register, onboarding |
| **Admin panel** | `http://localhost:3000/admin` | Requires `is_superadmin = true` |
| **Tenant dashboard** | `http://marco-dashboard.localhost:3000` | Demo tenant from seed |
| **Tenant booking PWA** | `http://marco.localhost:3000` | Public booking page |
| **Supabase Studio** | `http://localhost:54323` | DB explorer, SQL editor, auth |
| **Inbucket (email)** | `http://localhost:54324` | Catches all outbound emails locally |
| **Supabase API** | `http://localhost:54321` | Direct Supabase REST/auth calls |
| **PostgreSQL** | `postgresql://postgres:postgres@localhost:54322/postgres` | Direct DB connection |

#### Accessing tenant routes in local dev (two methods)

**Method A — Subdomain (recommended):** The proxy natively handles `*.localhost:3000`:
```
http://marco-dashboard.localhost:3000   → tenant dashboard
http://marco.localhost:3000             → tenant booking PWA
http://marco-app.localhost:3000         → tenant PWA (app shell)
```

**Method B — Query params (fallback if subdomains don't resolve):**
```
http://localhost:3000/?_tenant_slug=marco&_tenant_type=dashboard
http://localhost:3000/?_tenant_slug=marco&_tenant_type=landing
```
This fallback is built into `src/proxy.ts` for environments where `*.localhost` subdomains don't resolve.

## Section 5 — External Services in Local Dev

### Resend (email)

**What breaks without it:** Team member invitations (`/dashboard/team` → "Invita membro") will fail with an error. The invite record is created in the DB but the email is not sent, and `inviteTeamMember()` rolls back the invite.

**Local fallback:** Supabase's **Inbucket** (`http://localhost:54324`) catches all emails sent through Supabase Auth (password reset, magic links). However, Resend is used directly by the app code for team invitations — Inbucket does NOT intercept those.

**For local testing:** Set a real key. The free tier at [resend.com](https://resend.com) allows 100 emails/day with no credit card. You can send to your own email immediately (no domain verification needed for the test API key).

```
RESEND_API_KEY=re_your_actual_key_here
```

### Stripe (billing)

**What breaks without it:** Nothing in v1 MVP — Stripe is not integrated yet (keys are commented out in `.env.example`). The billing UI in the admin panel shows placeholder data.

**Local fallback:** Leave commented out. No action needed.

### Twilio (SMS/WhatsApp)

**What breaks without it:** SMS/WhatsApp notifications are not sent. The rest of the app (bookings, CRM, loyalty) works normally.

**Local fallback:** Leave commented out. For local dev the feature is not gated.

### Sentry (error monitoring)

**What breaks without it:** Errors are not reported to Sentry. They still appear in the terminal/browser console.

**Local fallback:** Leave commented out.

### PostHog (analytics)

**What breaks without it:** User analytics events are not tracked. No UI impact.

**Local fallback:** Leave commented out.

## Section 6 — Git Workflow

Based on `CLAUDE.md` and repo structure:

### Branch strategy

```bash
# Create a feature branch from main
git checkout main
git pull origin main
git checkout -b feature/my-feature-name

# Work, commit, push
git add .
git commit -m "feat: describe your change"
git push origin feature/my-feature-name
```

### Branch naming

Use kebab-case with a type prefix:
- `feature/staff-impersonation`
- `fix/banner-zindex`
- `chore/update-deps`

### Merging to main

`main` = production. Only merge when the feature is complete and tested. Vercel automatically deploys `main` to production.

### Vercel preview deploys

Every branch pushed to GitHub gets a Vercel preview URL automatically. The preview uses production Supabase (not local), so migrations must be applied to the remote project before testing on preview. Apply via:

```bash
supabase db push --db-url <production-db-url>
# or via Supabase dashboard: Migrations tab
```

## Section 7 — Known Issues Affecting Local Dev

### 1. `pnpm type-check` fails at root level

**Symptom:** Running `pnpm type-check` from the repo root errors with "missing script: type-check" because `apps/web/package.json` has no `type-check` entry.

**Workaround:**
```bash
pnpm --filter web exec tsc --noEmit
```

### 2. Shadow Mode shows admin profile instead of tenant owner

**From CLAUDE.md:** When an admin enters shadow mode (impersonating a tenant), pages that call `supabase.auth.getUser()` directly will return the admin's profile instead of the impersonated owner's profile.

**Impact on local dev:** If you test the admin shadow mode (`/admin` → enter a tenant), some dashboard data may incorrectly show the admin's info. This is a known architectural issue, not a setup problem.

**Workaround:** Use `resolveActiveProfile()` from `src/lib/tenant-context.ts` instead of calling `auth.getUser()` directly in dashboard pages.

### 3. Team invitation emails require a real Resend key

**Symptom:** "Errore nell'invio dell'email" toast when inviting staff.

**Fix:** Add a real `RESEND_API_KEY` to `.env.local`. See Section 5.

### 4. Seed fails if run from wrong directory

**Symptom:** `psql: error: supabase/seeds/01_base.sql: No such file or directory`

**Cause:** `seed.sql` uses `\i supabase/seeds/...` which is relative to the CWD.

**Fix:** Always run the seed command from the **repo root**:
```bash
# ✅ correct
psql postgresql://postgres:postgres@localhost:54322/postgres -f supabase/seed.sql

# ❌ wrong (from inside supabase/ dir)
cd supabase && psql ... -f seed.sql
```

### 5. `*.localhost` subdomains may not resolve on some systems

**Symptom:** `http://marco-dashboard.localhost:3000` shows "site can't be reached".

**Cause:** Some systems don't resolve `*.localhost` by default (mostly Linux).

**Fix:** Use the query-param fallback instead:
```
http://localhost:3000/?_tenant_slug=marco&_tenant_type=dashboard
```
This is a built-in fallback in `src/proxy.ts` (development only).
